argument-hint: "[--feature=<package-substring>] [--output-dir=<path>] [--list-only] [--max=<N>]"

## User Input

```text
$ARGUMENTS
```

Parse optional arguments:

| Argument | Default | Description |
|---|---|---|
| `--feature=<substring>` | _(all)_ | Filter to screens whose package contains this substring (e.g. `loans`, `creditcard`, `cashadvance`) |
| `--output-dir=<path>` | `.claude/screen-captures/` | Where to save the final PNGs |
| `--max=<N>` | _(unlimited)_ | Stop after N screens (useful for testing) |
| `--list-only` | `false` | Just list discoverable screens without rendering |
| `--stitch` | `false` | After rendering, stitch all PNGs into a single flow image |

---

## Overview

This skill discovers every `@Preview`-annotated Composable in the app, generates a temporary Paparazzi test for each one, renders them headlessly on the JVM, and collects the resulting PNGs. No device or emulator needed.

**Output**: A directory of PNG screenshots — one per screen/preview — representing what users actually see in the Android app.

---

## Step 0 — Validate Environment

**Android repo dependency**: This command requires `~/empower/empower-android` on branch `customer-agent/internal-composables`. Check that `~/empower/empower-android/empower-app/build.gradle` exists. If not, stop and tell the user: "This command requires ~/empower/empower-android on branch customer-agent/internal-composables."

Then `cd ~/empower/empower-android` before proceeding.

## Step 1 — Discover Screens

Search for all `@Preview` functions in the main source set:

```bash
grep -rn "@Preview" empower-app/src/main/java/finance/empower/android/app/ --include="*.kt" -l
```

For each file found, read it and extract:

1. **Package**: from the `package` declaration at the top of the file
2. **Preview function name**: the `@Composable` function annotated with `@Preview`
3. **Preview body**: the content of the preview function (the Composable call and its data construction)
4. **Visibility of the called Composable**: check whether the content Composable the preview calls is `private`, `internal`, or public

**Build a screen list** as a table:

| # | File | Package | Preview Function | Calls Composable | Visibility | Renderable? |
|---|---|---|---|---|---|---|

**Renderable criteria:**
- The Composable called by the preview must be `public` or `internal` (accessible from a test file in the same package)
- OR the preview function itself calls a public Composable (even if it wraps it with data)
- If the content Composable is `private`, mark as `SKIP — private` and note it in the summary

If `--feature` is provided, filter the list to files whose package contains the substring.
If `--max` is provided, truncate the list.
If `--list-only` is set, print the table and stop here.

## Step 2 — Generate Paparazzi Test Class

For all renderable screens, generate a **single** test class that contains one `@Test` method per screen. This is more efficient than one test class per screen because Paparazzi only needs to initialize once.

**Test file path:**
```
empower-app/src/test/java/finance/empower/android/app/CustomerAgentScreenshotTest.kt
```

**Template:**

```kotlin
// TEMPORARY — generated by /screenshot-screens, delete after capture
package finance.empower.android.app

import app.cash.paparazzi.DeviceConfig
import app.cash.paparazzi.Paparazzi
import finance.empower.android.design.system.TiltTheme
import org.junit.Rule
import org.junit.Test
// ... additional imports needed by preview data construction

class CustomerAgentScreenshotTest {

    @get:Rule
    val paparazzi = Paparazzi(
        deviceConfig = DeviceConfig.PIXEL_5,
        theme = "android:Theme.Material.Light.NoActionBar",
    )

    @Test
    fun screen_001_loanClosure() {
        paparazzi.snapshot("001_LoanClosure") {
            TiltTheme {
                // Body copied from @Preview function:
                LoanClosureScreen(
                    viewState = LoadingViewState.ContentWrapper(Content.DEFAULT),
                    onBackPressed = {},
                )
            }
        }
    }

    @Test
    fun screen_002_loansOfferCongrats() {
        paparazzi.snapshot("002_LoansOfferCongrats") {
            TiltTheme {
                LoansOfferCongratsContentA(amount = 250)
            }
        }
    }

    // ... one @Test per renderable screen
}
```

**Important rules for generating the test:**

1. **Use `paparazzi.snapshot("descriptive_name")`** — the string argument controls the output filename
2. **Copy the preview body exactly** — use the same data construction, callbacks (`= {}`), and parameters that the `@Preview` function uses
3. **Add all necessary imports** — for each preview's data types, ViewState wrappers, Content classes, R.string references, persistentListOf, etc. Collect imports from each source file.
4. **Handle visibility**: only include screens where the called Composable is accessible. The test is in package `finance.empower.android.app` so it can access:
   - Any `public` Composable in any subpackage (with import)
   - Any `internal` Composable in the same module
   - It CANNOT access `private` Composables — skip those
5. **If the preview calls a private Composable**, check if there's an equivalent public/internal Composable with the same parameters. If so, use that instead. If not, skip.
6. **Number each test** with a sequential prefix (001, 002, ...) for easy sorting of output files

## Step 3 — Build and Fix

Run the Paparazzi record task:

```bash
cd {REPO_ROOT}
./gradlew :empower-app:recordPaparazziDevelopmentDebug \
  --tests "finance.empower.android.app.CustomerAgentScreenshotTest" \
  2>&1
```

**If the build fails:**
1. Read the error output
2. Common issues:
   - **Missing import**: add the import and retry
   - **Private Composable access**: remove that test method and retry
   - **Missing `.DEFAULT` or `.FAKE`**: check if the data class has a companion; if not, construct minimal data manually
   - **Resource reference issues**: ensure R.string imports are correct
3. Fix the test file and re-run
4. If a single screen causes persistent failures, comment out that test method and note it in the report — don't let one bad screen block all others
5. Maximum 5 fix-and-retry cycles. After that, proceed with whatever screens compiled successfully.

## Step 4 — Collect Screenshots

After a successful build, Paparazzi writes PNGs to:
```
empower-app/src/test/snapshots/images/
```

The files are named based on the snapshot label:
```
finance.empower.android.app_CustomerAgentScreenshotTest_001_LoanClosure.png
finance.empower.android.app_CustomerAgentScreenshotTest_002_LoansOfferCongrats.png
...
```

Copy all generated PNGs to the output directory:

```bash
OUTPUT_DIR="{--output-dir or .claude/screen-captures/}"
mkdir -p "$OUTPUT_DIR"

# Copy and rename to cleaner filenames
for f in empower-app/src/test/snapshots/images/finance.empower.android.app_CustomerAgentScreenshotTest_*.png; do
  # Extract the descriptive name (everything after "CustomerAgentScreenshotTest_")
  name=$(basename "$f" | sed 's/finance.empower.android.app_CustomerAgentScreenshotTest_//' | sed 's/.png$//')
  cp "$f" "$OUTPUT_DIR/${name}.png"
done
```

Ensure the output directory is gitignored:
```bash
grep -qxF '.claude/screen-captures/' .gitignore 2>/dev/null \
  || echo '.claude/screen-captures/' >> .gitignore
```

## Step 5 — Stitch (optional)

If `--stitch` was passed, combine all PNGs into a single image for easy review.

Check ImageMagick availability:
```bash
which magick 2>/dev/null || which convert 2>/dev/null
```

If available, create a grid montage:
```bash
MAGICK=$(which magick 2>/dev/null || which convert)
$MAGICK montage "$OUTPUT_DIR"/*.png \
  -tile 3x -geometry 400x800+10+10 \
  -label '%f' -font Helvetica -pointsize 14 \
  "$OUTPUT_DIR/all_screens_montage.png"
```

If ImageMagick is not available, skip and note it in the report.

---

## Step 6 — Clean Up

Delete the temporary test file and generated snapshots from the Paparazzi directory:

```bash
rm -f empower-app/src/test/java/finance/empower/android/app/CustomerAgentScreenshotTest.kt
rm -f empower-app/src/test/snapshots/images/finance.empower.android.app_CustomerAgentScreenshotTest_*.png
```

Do NOT delete the copied PNGs in the output directory — those are the deliverable.
